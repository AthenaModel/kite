DESCRIPTION

This man page defines the ehtml(5) "Extended HTML" macro set, i.e., HTML extended with Tcl macros. When ehtml(n) is registered with an instance of macro(n), the instance will translate ehtml(5) input into plain HTML.

The ehtml(5) format is the basis for more specific formats, including both kitedoc(5) and manpage(5), the format used to write this man page. The basics of the format are documented here.

BASIC SYNTAX

The majority of the content of a ehtml(5) document is standard HTML body text. Macros are entered in double angle brackets, like so: 

<mymacro>

Some macros occur in pairs; for example, change logs are created using the <changelog> macros. 

<changelog>
    .
    .
    .
</changelog>

MACROS

ehtml(5) input may use any of the following macros; formats like manpage(5) will add additional macros.

Control Macros

The following macros are used to control processing by the kitedocs(n) document processors.

<nopara>...</nopara>

These tags are for use with ehtml(5) processors that use htmltrans(n)'s automatic paragraph detection filter (e.g., manpage(n) and kitedoc(n)).

The tags disable automatic paragraph detection for the content they contain. The content is treated like any other "opaque" content (e.g., the HTML <head> element); structure and tag names are checked, but no paragraph tags are inserted.

The <nopara> and </nopara> tags themselves are stripped during processing by htmltrans(n).

If used with other ehtml(5) processors, the tags will be accepted but will simply pass through into the output.

HTML Equivalent Macros

The following macros are used in place of their HTML equivalents. They take no arguments; any formatting is done using CSS provided by the document format.

<blockquote>...</blockquote>
<p>............</p>
<li>...........</li>
<ol>...........</ol>
<table>........</table>
<tr>...........</tr>
<th>...........</th>
<td>...........</td>
<ul>...........</ul>
<br>
These macros simply expand to their HTML equivalents.
<img ?attr=value...?>
Returns an HTML "<img>" tag. Any arguments are passed through as is.

HTML Style Macros

Macros relating to the style of displayed text.

<b>........</b> or <b words...>
<code>.....</code> or <code words...>
<em>.......</em> or <em words...>
<h1>.......</h1> or <h1 words...>
<h2>.......</h2> or <h2 words...>
<h3>.......</h3> or <h3 words...>
<h4>.......</h4> or <h4 words...>
<h5>.......</h5> or <h5 words...>
<h6>.......</h6> or <h6 words...>
<i>........</i> or <i words...>
<pre>......</pre> or <pre words...>
<strong>...</strong> or <strong words...>
<sub>......</sub> or <sub words...>
<sup>......</sup> or <sup words...>
<tt>.......</tt> or <tt words...>

Called with no arguments, these macros simply expand to the matching HTML tags. Alternatively, each can be called with some number of arguments, which are returned wrapped in matching tags. In other words, in ehtml(5) input the following items are equivalent: 

<i>italic text</i>

vs. 

<i italic text>

The second form is easier to type. Note that these macros do not expand their arguments.

Cross-Reference Macros

The following macros are used to define references and to make cross-reference links.

<xref id ?anchor?>

The <xref> macro creates hyperlinks. Generally speaking, id identifies the page to link to; anchor is the link text. If anchor isn't specified, then the id will be used as the link text.

The <xref> macro recognizes three kinds of page IDs, in this order:

  • IDs defined using <xrefset>.
  • Man page references, e.g., "name(section)". This creates a link to docroot/mansection/name.html>, where docroot is a relative path set via the ehtml(n) -docroot configuration option.
  • Document references. If the ID appears to end with a file name with a typical document extension (as determined by the ehtml(n) -doctypes configuration option), it is treated as a path relative to docroot.

In the latter two cases, the ID may begin with "project:". In this case, it's assumed that "project" is the name of a sibling Kite project; and it's further assumed that this project will usually be checked out into the same parent directory as the current project. Consequently, the man page or document reference is relative to the project's docs/ directory, e.g., the path is preceded by "docroot/../../project/docs/".

<xrefset id anchor url>

This macro defines a cross-reference, which can be used by <xref> to create hyperlinks. The id identifies the cross-reference to <xref>; it should be chosen so as not to conflict with other <xref> identifiers. The anchor is the link text that will be used if no link text is specified in the <xref> macro. The url is, of course, the URL to link to.

The macro returns a link to the url with the given anchor text, so that it can be used in a list of references for display.

Note that <xrefset> can appear anywhere in the input; it does not need to proceed the first <xref> to the id.

For example, the following macro defines "google" as a cross-reference ID. 

<xrefset google "Google" "http://www.google.com">
    .
    .
    .
To find something, use <xref google>....

Ordered and Unordered List Macros

In addition to the standard <ol> and <ul> lists, ehtml(5) defines versions of these lists that expect list items to be full; consequently, the items are displayed with blank lines between them:

<olp>...</olp>
<ulp>...</ulp>
These macros enclose ordered and unordered lists whose <li>...</li> items should have blank lines between them.

Definition List Macros

The following macros are used to add definition lists to a document. A definition list displays a term followed by a definition of a term; usually, the definition is indented beneath the term.

<deflist ?args?>...</deflist ?args?>

These tags contain a definition list. The individual definitions are added using the <def>, <defitem>, or <defopt> macros.

Definition lists are frequently nested; the optional args can be used to relate the <deflist> to its </deflist>. In addition, <defitem> accumulates lists of items by the lists in which they appear, for retrieval via <itemlist>. 

<section COMMANDS>

<itemlist commands>

<deflist commands>
    .
    .
    .
</deflist commands>

Some document formats may define <def*> tags with additional semantics, for use in <deflist>s.

<def text>

This macro begins a definition within a <deflist>. The text is the text to display; it may contain embedded style macros.

The def macro is followed by the text that defines the named item.

<defitem tag text>

This macro begins an item definition within a <deflist>, that is, a definition of a code item: a Tcl command, a program, a URL, or whatever it might be.

The tag is a short string which identifies the item, such as a command's name; it should be unique within the document. It becomes the item's HTML anchor, and is also used with <iref> to reference the item from within the same document.

The text is the text that's actually displayed in the list; it usually includes more information, such as a command's full signature. It may contain embedded style macros.

The defitem macro is followed by the text that defines the named item. For example, 

<defitem mycommand {mycommand <i value> ?<i optionalValue>?}>

This command does nothing in particular, being merely a pedagogical
example.<p>;

Note that the tag won't necessarily be same as the first token in the text; when documenting subcommands of a Tcl object, for example, the object usually appears as a variable reference in the first spot: 

<defitem mysubcommand {$object mysubcommand <i value>}>

Each of the macro definitions in this section is produced by defitem.

<defopt text>

Programs often have options; Tcl commands and objects often have options as well. When <defopt> appears in a definition list, it defines an option belonging to the item defined by the immediately preceding <defitem>. It's assumed that the option is the first token in text, which may contain embedded style macros. For example, 

<deflist commands>
    . . .
<defitem mycommand {mycommand ?< options...>?}>

This command has the following options:<p>

<deflist mycommand options>

<defopt {-this <i value>}>

Description of the the option...
    . . .
</deflist mycommand options>
. . .

</deflist commands>
<itemlist ?listname?>

By default, this macro expands to a list of the items defined in this document. It's usually used in the "SYNOPSIS" section of a manpage(5) document. The list of macros in the SYNOPSIS section of this man page was produced by <itemlist>.

If an item has options defined with <defopt>, they will appear, slightly indented, just below the item in the <itemlist>.

If listname is given, the output is confined to those items defined in the named <deflist> and any lists nested within it.

<iref item...>

The name "iref" stands for "item reference"; this macro creates a hyperlink to the definition of item in the same man page. For example, 

<iref defitem>

expands to a link to defitem.

If the item name has multiple tokens, there is no need to quote them; <iref>'s arguments are accumulated into a single item name.

<itag name>
This is simply an <iref> to macro name; it creates a hyperlink in typewriter font with angle brackets.

Topic List Macros

A topic list is a table with two columns, one for topics and one for descriptions. The column headers default to "Topic" and "Description", but can be changed.

<topiclist ?h1? ?h2?>...</topiclist>
Brackets a <topic> list. The h1 and h2 values are the column header strings; they default to "Topic" and "Description" respectively.
<topic text>...</topic>
Adds a topic to a <topiclist>. The topic is displayed in two columns, the topic text on the left and the discussion between the two tags on the right.

Examples, Listings, and Marks

<example>...</example>
These macros are used to bracket pre-formatted examples. The examples are displayed in a box with a different background than the rest of the page.
<listing ?firstline?>...</listing>

These macros are used to bracket pre-formatted code listings. Line numbers are added, so that the surrounding text can refer to specific lines. The listing is displayed in a box with a different background than the rest of the page.

By default, the first line is numbered 1; set firstline to pick a different number.

<mark symbol>

Defines a marker in an <example> or <listing>. The marker displays the symbol in a small rounded box with a red background, and is used to point out lines of interest for discussion in the following paragraphs.

Use <bigmark> to display the same marker as an <topic> in a <topiclist>.

<bigmark symbol>
Creates a larger version of a <mark>; the larger version looks better as the <topic> in a <topiclist>.

Change Log Macros

The following macros are used to add a change log to a document.

<changelog>...</changelog>
Defines a change log table. The content between the beginning and end macro should consist of <change> entries.
<change date status initiator>...</change>

Defines a single change entry within a <changelog>. The change is as of the specified date and was made by the initiator; the status can indicate whether the change is an addition, a deletion, or whatever the user likes. The Extended HTML input between the beginning and end tag is expanded as usual, and is used as the description of the change.

For example, 

<changelog>
<change 10/01/2008 New whd>
Initial draft.
</change>
</changelog>

Procedure Macros

The following macros are used to document procedures consisting of a number of steps.

<procedure>...</procedure>

Defines a procedure: instructions for performing a set of commands at a command line. Each step consists of a description of what is to be done, and of an example of the commands to enter. The procedure is formatted as a table of numbered steps, with each step's description in the left-hand column and example in the right-hand column.

The content between the beginning and ending macros should consist of <step> entries.

<quote text>
Quotes angle brackets and ampersands in the text for HTML display.
<step> description </step/> example </step>

Defines a single step within a <procedure>. The description is Extended HTML text saying what is to be done, and the example is Extended HTML text giving an example of how to do it.

For example, 

<procedure>

<step>
Edit the document.
</step/>
<pre>
$ emacs mydoc.ehtml
</pre>
</step>
    .
    .
    .

</procedure>

Miscellaneous Macros

<hrule>
Produces a horizontal rule across the page.
<lb>
Returns the left-hand macro bracket, e.g., "<".
<link url ?anchor?>

Creates a link to the specified url, displaying the specified anchor text. If anchor is omitted, the url itself is displayed.

If the page will be linked multiple times, it might be better to define a cross-reference using <xrefset>.

<nbsp text>
After trimming leading and trailing whitespace and normalizing internal whitespace to single blanks, <nbsp> replaces all remaining whitespace with non-breaking spaces.
<rb>
Returns the right-hand macro bracket, e.g., ">".
<tag arglist>
Formats a macro tag in angle-brackets. If the arglist is given, it is expanded and included in the tag.
