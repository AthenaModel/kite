SYNOPSIS

  • <manpage nameList description>
  • </manpage>
  • <section name>
  • <subsection name>
  • <version>
  • <mktree id>...</mktree>

DESCRIPTION

This man page defines the manpage(5) man page format, a flavor of "Extended HTML", i.e., an HTML-like format extended with Tcl macros. The basic syntax and capabilities of Extended HTML are defined in the ehtml(5) man page. This man page describes an extension to the basic format, as an aid to writing man pages.

The manpage(n) module translates man pages in manpage(5) format into HTML.

Automatic Transformations

The manpage(n) processor uses htmltrans(n) to transform the output of the macro processor. In particular, it supports:

  • Automated closure of <li> and similar tags.
  • Automatic paragraph detection and insertion of <p>...</p> tags. Essentially, blank lines in prose force a paragraph break. There is no need to insert the paragraph tags by hand.

MACROS

manpage(5) man pages may use any of the following macros, as well as those defined in the ehtml(5) man page. Note that these macros are simply Tcl commands; you can define your own macros, as described in ehtml(5), and use them in your man pages.

<manpage nameList description>

This macro should be the first in the man page, as it expands to the HTML header and also the "NAME" section of the man page.

The macro has two forms. In the first form, nameList is just the name of the man page; description is a short one-line description of the documented library, file format, or program: 

<manpage manpage(5) "Kite Man Page Format">

The second form is used when the man page documents a submodule of a larger library. In this case, nameList is a list containing two entries, the name of the parent man page and the name of this man page. For example, the man page for kitedocs(n)'s marsmisc(n) module begins with this macro: 

<manpage {kitedocs(n) kitedoc(n)} "Kite Document Formatter">
</manpage>

This macro should be the last in the man page, as it expands to the HTML footer. It also adds a timestamp for when the page was translated.

For example, 

</manpage>
<section name>

Begins a new section. By convention, man page section names are displayed in all caps; consequently, name is converted to all caps when needed.

Every section is included in a table of contents at the beginning of the generated man page.

The name can be almost anything; however, see MAN PAGE SKELETON for a list of the sections found in most man pages.

<subsection name>

Begins a new subsection under the current section. By convention, man page subsection names are displayed in mixed case. Subsections are included in the table of contents, indented under their parent section.

The subsection name can be almost anything, but it should be unique within the man page.

<version>
Returns the project version number, as provided to manpage(n)'s format subcommand.
<mktree id>...</mktree>

This tag defines an unordered "<ul>" list that uses the "mktree" Javascript package for turning nested lists of links into dynamic lists that can be opened and closed. The Javascript code and the necessary CSS styles are included in the HTML output automatically.

The id is used as the value of the "id" attribute of the <ul> tag, which is generated like this: 

<ul class="mktree" id="id">

MAN PAGE SKELETON

The sections included in a man page depend on just what you're documenting. That said, here's a fairly typical skeleton: 

<manpage somelib(n)>

<section SYNOPSIS>

Put how to load this library here.  Then, put the itemlist:

<itemlist>

<section DESCRIPTION>

A general description of the package.

<section COMMANDS>

If this were a C library, this might be "FUNCTIONS" instead.

<deflist commands>
     .
     .
     .
</deflist commands>

<section EXAMPLES>

Put examples here, assuming they didn't fit better under one of the
items listed above.

<section AUTHOR>

Your name.

<section HISTORY>

Any relevant historical information, like "This package was developed
as a replacement for <xref soandsolib(n)>...."

</manpage>
